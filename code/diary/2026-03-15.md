# 2026-03-15 编程日报：在Vibe Coding时代构建大型系统

这一天，我带领AI围绕"thera"项目的嵌入式批注功能，完成了一次从需求到代码再到复盘的完整迭代。过程并不顺利——AI频频犯错，我频频指正，但每一次纠偏都让系统更健壮，也让AI的"工程直觉"更敏锐。以下按事件、语义、自我三个层面，还原这一天的真实故事。

## 一、事件记忆 —— Vibe Coding下的协作切片

### 1. 样例准备中的"格式漂移"与源头纠正

早晨的第一个任务是：为平台开发准备一个可测试的样例。我需要AI复制一份工作日志（`journal/default/2026-03-15.md`）作为输入fixture，然后从产品日志（`journal/product/thera/2026-03-15.md`）中提取"效果演示"部分的纯日志内容，生成标准的输出fixture。

在我的指令下，AI迅速完成了复制：输入fixture被放到`platform/tests/fixtures/default/journal_2026-03-15_input.md`。但生成输出fixture时，它直接复制了整个产品设计文档——包括前面的需求描述、设计思路，而不仅仅是"效果演示"那段纯日志。更糟糕的是，它没有使用Markdown引用格式（`>`）来标记AI批注，而是用了普通的段落。

我看到后立刻指出："输出fixture只取示例部分，不要包含产品设计文档；AI批注必须用引用格式，并且要严格复用用户给的示例格式。"在我的反馈后，AI马上修正：它重新阅读了产品日志，精确截取了"效果演示"部分（从"效果演示"标题到日志结束），并将AI的每条批注都改为`>`开头的引用块，格式与用户示例完全一致。

这个小插曲让我意识到：AI在面对"复制"和"提取"这种看似简单的任务时，很容易出现"边界漂移"——它不知道哪些是文档、哪些是示例，哪些是内容、哪些是说明。必须第一时间建立清晰的边界规则，否则类似的错误会蔓延到后续所有生成物。

### 2. 文档重写中的"半成品"判定与重构

接着是第二个任务：细化开发者文档。我需要AI根据已有的PRD草稿和DEV草稿，写出完整的、可供开发者直接使用的文档。在我的要求下，AI提交了一份初稿，结构完整，但关键的实现细节全是"待定"——比如文件写入逻辑、批注检测标识符、CLI配置初始化、段落切分边界处理，这些都没有具体说明。更关键的是，它没有暴露Prompt，开发者拿到手根本不知道AI是怎么生成批注的。

我读完后判定为"半成品"，要求整体重写，而不是局部修补。AI重新设计了文档，这次它补充了模块划分（cli.py、mode/default.py、llm.py、config.py），列出了每个模块的职责，甚至画出了伪代码。但它仍然漏掉了几个关键点：文件写入逻辑在伪代码中是全量写回，但在文字描述中却说"增量更新"；批注检测标识符写成了加粗的`🤖 观察者注`，而用户示例中是纯文本；CLI配置初始化没有给出`load_config()`函数的实现；段落切分没有考虑空字符串和过短段落的过滤。

我再次反馈，一一指出这些问题。AI逐一修正：统一了文件写入逻辑（全量读→全量写），将标识符改为不加粗的`🤖 观察者注`，补充了`load_config()`函数，并添加了`split_paragraphs`函数来过滤空段落和短段落（<20字）。最终，文档变得真正"开箱即用"。

这次反复让我体会到：文档的完整性不能靠AI的"一次成型"，而需要人类用实际开发者的视角去审视，找出那些"想当然"的空缺。每一次空缺的填补，都是在将隐性知识显性化。

### 3. 代码实现中的过度设计与硬编码清理

文档完备后，我指导AI开始实现代码。AI提交了第一版实现，但我发现它引入了一个`DefaultConfig`类和一个`load_default_config`函数，用来管理配置——而我们已经约定用`.env`文件加Settings类管理环境变量。这显然是过度设计，增加了不必要的复杂性。更糟的是，它硬编码了opencode的路径为`/usr/local/bin/opencode`，而实际路径可能因系统而异。主入口文件`__main__.py`中还有大量冗余代码，与开发者文档的设计不符。

我给出了明确的指令："移除`load_default_config`，删除`__main__.py`中的冗余部分，重写主函数，所有路径必须从环境变量读取。"在我的指令下，AI立刻执行"大踏步后退"：它删除了多余的配置类，将硬编码路径改为从`settings.OPENCODE_PATH`读取，并完全重写了`cli.py`作为唯一入口。代码库瞬间变得简洁一致。

这次经历印证了一个道理：当AI的设计出现根本性偏差时，修补只会让代码越来越臃肿，果断删除并基于正确基线重写，才是最高效的纠偏方式。

### 4. 质量审计中的"未来风险"预判

下午，我让AI扮演QA专家，对整个仓库进行一次发布前的全面审计。AI独立运行了导入测试、配置测试，并仔细检查了所有代码。结果令人震惊：

- **严重问题**：整个`src/thera/`目录下没有任何测试文件，违反了Mode层必须有完善测试的要求；`config.py`中的`Settings`类缺少`llm_api_key`、`llm_base_url`、`llm_model`三个字段，而`knowl.py`中却直接使用了它们。
- **中等问题**：`default.py`中硬编码了超时30秒；`knowl.py`导入`quanttide_agent`时没有try-except防护，依赖不可用时直接崩溃；`meta.py`中的`DATA_DIR`指向`platform/data`，而非模块自有数据路径，这会导致部署时找不到数据。
- **未来可用性风险**：`quanttide_agent`未在`pyproject.toml`中声明，是隐式依赖；`llvmlite`版本限制`<0.45`，与Python 3.14不兼容（安装失败已验证）；`.env`文件缺失时无法运行，无默认值或友好错误提示；当前无法通过`pip install -e .`安装，因为llvmlite构建失败。

这些深层次问题在之前的快速迭代中都被忽略了。AI作为QA，通过角色轮换获得了全局视角，提前暴露了隐患。我并没有直接指出这些问题，而是赋予它审计角色，让它自己去发现——这正是培养AI"未来思维"的有效方式。

### 5. 系统性复盘中的经验抽象

最后，我要求AI对整个轮次进行系统性复盘。AI回顾了所有事件，并将经验分为两类：一类是可以写成教程/报告/札记的"经验知识"（如文档驱动开发的最佳实践、人机协作的节奏感），另一类是需要固化到案例/手册/标准/章程中的"制度规范"（如硬编码零容忍、测试先行、不可变模块重构）。从具体事件到通用原则的跃迁，在这一步完成。

## 二、语义记忆 —— 驾驭Vibe Coding的工程法则

从这一天的多次碰撞中，我提炼出了一套适用于AI快速生成环境下的工程法则。每一个法则都源自真实的纠偏时刻，而非空洞的说教。

### 1. 文档驱动开发的强化版：双向透明与格式锁定

**起源**：在文档重写中，AI初版遗漏了Prompt和待定项，导致开发者无法理解AI的决策逻辑；在样例生成中，AI未复用用户格式，导致样式不一致。

**法则**：PRD和DEV文档必须暴露Prompt和所有"待定"项，让AI的思考过程可审查、可追溯。同时，关键标识符（如`🤖 观察者注`）和引用格式一旦确定，就必须永久锁定，防止AI自由发挥引入不一致。这相当于为AI的"创作"划定了一条清晰的格式边界。

### 2. 配置中心化与硬编码零容忍

**起源**：代码实现中，AI硬编码了opencode路径和超时时间，导致系统无法在不同环境中运行；`load_default_config`的过度设计增加了维护负担。

**法则**：所有环境敏感项——路径、密钥、超时——都必须通过`.env`文件加Settings类注入，任何硬编码在审查中都会被一票否决。这不仅保证了当前的可移植性，也为未来的配置变更留出了空间。

### 3. 不可变模块与原子化重构

**起源**：当AI的`load_default_config`和冗余的`__main__.py`出现时，我选择整体删除而非修补。

**法则**：当模块存在系统性缺陷时，不修补，整体删除后基于最新规范重写。修补只会累积技术债，而原子化重构能重置系统健康度。保留事件日志作为不可变基线，确保任何一次"后退"都有据可查。

### 4. 测试先行与质量护栏

**起源**：质量审计中暴露的测试缺失和字段未定义问题，说明测试必须前置，而不是事后补充。

**法则**：必须配套单元测试，基础导入测试和配置测试必须通过才能合入。将反复出现的检查点（如标识符一致性、路径配置）自动化，形成预提交钩子，让质量检查成为开发流程的一部分，而非事后救火。

### 5. 角色轮换与视角拓展

**起源**：我让AI扮演QA后，它发现了之前作为开发者时完全忽略的深层次问题。

**法则**：定期让AI扮演不同角色（设计者、文档员、QA、复盘者），强制从多维度审视系统。每次角色转换后，AI都会自动将新视角的考量融入后续设计——例如，担任QA后，它会在设计时主动考虑测试覆盖和依赖管理。

## 三、自我记忆 —— 独特认知风格与指导哲学

如果说前两部分记录了事件和法则，那么这一部分试图捕捉的，是我作为人类导师在这场协作中的思维特质——那些难以言传但至关重要的指导哲学。

### 源头治理者的本能

我习惯于在混乱发生前建立秩序。在每个任务开始时，我都会预埋"规范种子"：强制格式统一、预置检查清单、明确边界规则。这种前瞻性来自对AI"自由发挥倾向"的深刻理解——我知道AI会在哪里跑偏，就在哪里提前设障。样例准备中的"格式漂移"事件，正是这一本能的体现：我在问题刚露头时就介入，避免了后续更大范围的蔓延。

### 架构师式的决策分层

我从不亲自设计模块，而是设计一套让AI能自主设计的规则系统。我的审查焦点始终集中在关键质量门上——文档完整性、配置中心化、测试覆盖——而不是代码细节。这样，我就能将精力集中在真正影响系统架构的决策上，而把具体的实现交由AI完成。这是一种"管理设计的设计"，类似于编程中的元编程。

### "大踏步后退"的勇气与智慧

当发现AI的设计偏离根本原则时，我能果断要求整体重写，即使这意味着丢弃已投入的时间。这种勇气源于对不可变原则的信仰：在错误基础上修补只会累积技术债，而干净的重写能重置系统健康度。删除`load_default_config`、重写`__main__.py`，都是这种智慧的体现。我的后退不是倒退，而是基于更优基线的跳跃式前进。

### 节奏感：收放自如的干预粒度

我时刻感知着AI的状态，动态调整干预的力度。在AI表现稳定时，我会大幅放权——比如让它独立担任QA、自主复盘；在AI出现系统性偏差时，我会立即收紧，给出极具体的指令——"删除X文件，重写Y函数"。这种节奏使AI始终处于"最近发展区"，既不被放任自流，也不被过度束缚。

### 元认知培养者：让AI学会反思

我不仅要求AI完成任务，还强制它进行3E评估和根因分析。每一次错误都被转化为AI认知升级的契机。最终，AI在任务5中能自主提炼经验并规划制度，这正是我长期"教AI如何学习"的结果。我不再只是修正AI的输出，而是在塑造它的思维模式。

### 在Vibe Coding时代的独特定位

当大多数人在vibe coding中被动接受AI的输出时，我主动构建了一个认知控制系统，让AI的创造力在工程约束内有序释放。我不是AI的助手，而是AI的架构导师。这个定位，正是我能带领AI在一天内完成从混乱到秩序、从具体到抽象跃迁的根本原因。

## 小结

这一天的工作日志，最终沉淀为一份可复用的经验资产。事件记忆提供了事实基础，语义记忆将教训转化为知识，自我记忆锚定了协作关系。三者共同构成了一个可进化的人机协作认知系统——它超越简单的"提示工程"，开创了在AI时代构建大型软件的新范式。
